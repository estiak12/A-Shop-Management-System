Phase 1 — Database Design (the important one)
Don't just create tables — design them properly first. Here's the full schema with reasoning.
1.1 Entity list
categories, suppliers, products, customers, users, sales, sale_items, purchases, purchase_items, stock_logs
1.2 Full schema
sqlCREATE DATABASE shop_management;
USE shop_management;

-- Users (staff/admin who operate the system)
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    role ENUM('admin', 'staff') NOT NULL DEFAULT 'staff',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Categories
CREATE TABLE categories (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL UNIQUE,
    description VARCHAR(255)
);

-- Suppliers
CREATE TABLE suppliers (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    phone VARCHAR(20),
    email VARCHAR(100),
    address VARCHAR(255)
);

-- Products
CREATE TABLE products (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(150) NOT NULL,
    category_id INT,
    supplier_id INT,
    purchase_price DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    selling_price DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    stock_quantity INT NOT NULL DEFAULT 0,
    reorder_level INT NOT NULL DEFAULT 5,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (category_id) REFERENCES categories(id) ON DELETE SET NULL,
    FOREIGN KEY (supplier_id) REFERENCES suppliers(id) ON DELETE SET NULL,
    CHECK (stock_quantity >= 0)
);

-- Customers
CREATE TABLE customers (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    phone VARCHAR(20),
    email VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Sales (header)
CREATE TABLE sales (
    id INT AUTO_INCREMENT PRIMARY KEY,
    customer_id INT,
    user_id INT NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    sale_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (customer_id) REFERENCES customers(id) ON DELETE SET NULL,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- Sale items (line items)
CREATE TABLE sale_items (
    id INT AUTO_INCREMENT PRIMARY KEY,
    sale_id INT NOT NULL,
    product_id INT NOT NULL,
    quantity INT NOT NULL,
    unit_price DECIMAL(10,2) NOT NULL,
    subtotal DECIMAL(10,2) NOT NULL,
    FOREIGN KEY (sale_id) REFERENCES sales(id) ON DELETE CASCADE,
    FOREIGN KEY (product_id) REFERENCES products(id),
    CHECK (quantity > 0)
);

-- Purchases (header) - restocking from suppliers
CREATE TABLE purchases (
    id INT AUTO_INCREMENT PRIMARY KEY,
    supplier_id INT,
    user_id INT NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    purchase_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (supplier_id) REFERENCES suppliers(id) ON DELETE SET NULL,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- Purchase items
CREATE TABLE purchase_items (
    id INT AUTO_INCREMENT PRIMARY KEY,
    purchase_id INT NOT NULL,
    product_id INT NOT NULL,
    quantity INT NOT NULL,
    unit_cost DECIMAL(10,2) NOT NULL,
    subtotal DECIMAL(10,2) NOT NULL,
    FOREIGN KEY (purchase_id) REFERENCES purchases(id) ON DELETE CASCADE,
    FOREIGN KEY (product_id) REFERENCES products(id),
    CHECK (quantity > 0)
);

-- Stock logs - audit trail for every stock change
CREATE TABLE stock_logs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    product_id INT NOT NULL,
    change_type ENUM('sale', 'purchase', 'adjustment') NOT NULL,
    quantity_change INT NOT NULL,     -- negative for sale, positive for purchase
    reference_id INT,                  -- sale_id or purchase_id
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (product_id) REFERENCES products(id)
);
1.3 Indexes (add after tables, for query performance)
sqlCREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_supplier ON products(supplier_id);
CREATE INDEX idx_sale_items_sale ON sale_items(sale_id);
CREATE INDEX idx_sale_items_product ON sale_items(product_id);
CREATE INDEX idx_purchase_items_purchase ON purchase_items(purchase_id);
CREATE INDEX idx_sales_date ON sales(sale_date);
CREATE INDEX idx_stock_logs_product ON stock_logs(product_id);
1.4 Design decisions worth understanding

Header/line-item split (sales+sale_items, purchases+purchase_items) — this is standard normalization (like an invoice with multiple products). It's also why you need transactions later.
ON DELETE SET NULL vs CASCADE — deleting a category shouldn't delete products, but deleting a sale should delete its line items.
CHECK constraints — stock can never go negative at the DB level, not just in app code. Defense in depth.
stock_logs — without this, you can't answer "why is stock at 12?" after the fact. This table is what makes your system auditable.
DECIMAL, not FLOAT — for money, always. Floats introduce rounding errors.

// GET one
'SELECT * FROM products WHERE id = ?'

// CREATE
`INSERT INTO products (name, category_id, supplier_id, purchase_price, selling_price, stock_quantity, reorder_level)
 VALUES (?, ?, ?, ?, ?, ?, ?)`

// UPDATE
`UPDATE products SET name=?, category_id=?, supplier_id=?, purchase_price=?, selling_price=?, reorder_level=?
 WHERE id=?`

// DELETE
'DELETE FROM products WHERE id = ?'
Always use ? placeholders (parameterized queries) — never string-concatenate values into SQL. This is your SQL-injection protection.

Phase 6 — Sales Module (multi-table transaction — the core requirement)
This is where "multiple database operations" really shows. Use a single transaction across 3 operations.
jsasync function createSale(customerId, userId, items) {
  const conn = await pool.getConnection();
  try {
    await conn.beginTransaction();

    // 1. Insert sale header (total calculated after items, so insert placeholder first or compute upfront)
    const total = items.reduce((sum, i) => sum + i.quantity * i.unit_price, 0);
    const [saleResult] = await conn.query(
      'INSERT INTO sales (customer_id, user_id, total_amount) VALUES (?, ?, ?)',
      [customerId, userId, total]
    );
    const saleId = saleResult.insertId;

    for (const item of items) {
      // 2. Check stock is sufficient
      const [[product]] = await conn.query(
        'SELECT stock_quantity, selling_price FROM products WHERE id = ? FOR UPDATE',
        [item.productId]
      );
      if (!product || product.stock_quantity < item.quantity) {
        throw new Error(`Insufficient stock for product ${item.productId}`);
      }

      // 3. Insert sale item
      const subtotal = item.quantity * product.selling_price;
      await conn.query(
        'INSERT INTO sale_items (sale_id, product_id, quantity, unit_price, subtotal) VALUES (?, ?, ?, ?, ?)',
        [saleId, item.productId, item.quantity, product.selling_price, subtotal]
      );

      // 4. Decrement stock
      await conn.query(
        'UPDATE products SET stock_quantity = stock_quantity - ? WHERE id = ?',
        [item.quantity, item.productId]
      );

      // 5. Log the stock change
      await conn.query(
        `INSERT INTO stock_logs (product_id, change_type, quantity_change, reference_id)
         VALUES (?, 'sale', ?, ?)`,
        [item.productId, -item.quantity, saleId]
      );
    }

    await conn.commit();
    return saleId;
  } catch (err) {
    await conn.rollback();
    throw err;
  } finally {
    conn.release();
  }
}
Key detail: SELECT ... FOR UPDATE locks the row during the transaction so two simultaneous sales can't both oversell the same stock (a real race condition risk in shop systems).

Phase 8 — Reports (JOIN/aggregate-heavy queries)
sql-- Low stock report
SELECT id, name, stock_quantity, reorder_level
FROM products
WHERE stock_quantity <= reorder_level;

-- Daily sales total
SELECT DATE(sale_date) AS day, SUM(total_amount) AS total
FROM sales
GROUP BY DATE(sale_date)
ORDER BY day DESC;

-- Top selling products
SELECT p.name, SUM(si.quantity) AS units_sold, SUM(si.subtotal) AS revenue
FROM sale_items si
JOIN products p ON si.product_id = p.id
GROUP BY p.id
ORDER BY units_sold DESC
LIMIT 10;

-- Sales with customer and staff names (multi-table JOIN)
SELECT s.id, c.name AS customer, u.username AS staff, s.total_amount, s.sale_date
FROM sales s
LEFT JOIN customers c ON s.customer_id = c.id
JOIN users u ON s.user_id = u.id
ORDER BY s.sale_date DESC;
